# v14.8.5.038 — runtime-host-audit-observability

## 1. Cel

Dodać trwały audit trail dla przepływu runtime-host: accept, repair, reject, duplicate, replay, conflict. `.038` przygotowuje fundament pod `.040`, gdzie transport MCP może powodować retry/timeout i wymaga idempotency.

## 2. Zakres

- `host_bridge_audit` jako schema/utility.
- Idempotency key dla finalizacji.
- Dedupe/replay policy.
- CLI do odczytu ostatnich zdarzeń audytu.
- Minimalizacja danych: hash + metadata zamiast duplikacji raw text, gdzie możliwe.

## 3. Proponowane moduły

## 5. Idempotency

`idempotency_key = sha256(turn_id + trace_id + finalization_contract_sha256 + original_text_sha256)`.

| Przypadek | Wynik |
|---|---|
| ta sama finalizacja, ten sam tekst | `duplicate_same_payload`, zwrócić zapisany wynik |
| ten sam turn/trace, inny tekst | `conflict`, reject |
| retry po timeout | safe replay |
| brak audytu | fallback JSONL audit + warning |

## 6. Testy

accept -> audit record; repair -> audit record z violation; reject -> audit record z error_code; duplicate same payload -> idempotent success; duplicate different payload -> conflict; audit survives process restart.

## 7. CLI

```powershell
python -X utf8 main.py --host-bridge-audit-tail --limit 20
python -X utf8 main.py --host-bridge-audit-find --turn-id <id>
```

## 8. Rollback

Audit ma być add-only; rollback kodu nie usuwa istniejących rekordów. Jeśli DB migration zawiedzie, fallback do JSONL audit w `workspace_runtime/`.
