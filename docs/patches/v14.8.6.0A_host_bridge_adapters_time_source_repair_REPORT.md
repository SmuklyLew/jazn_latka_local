# v14.8.6.0.A — host bridge, adaptery i resolver czasu

Wersja: `v14.8.6.0.A`
Nazwa wydania: `dokladne-naprawy-host-bridge-adaptery-time-source-i-drobne-potkniecia`
Gałąź robocza: `work/v14.8.6.0A-host-bridge-adapters-time-source-repair`

## Błąd

Runtime prawidłowo odrzucał szablon lub tekst handlera jako dynamiczną wypowiedź Łatki, ale w trybie ChatGPT kończył turę tekstem `RuntimeTurnTruthGate / model_guided_speech_unavailable`. Ten tekst był prawdomówny dla `null_model_adapter`, lecz dla `chatgpt_runtime_adapter` oznaczał niedomknięty protokół: właściwą następną fazą była generacja widzialnej odpowiedzi przez host ChatGPT.

Dodatkowo:

- `final_visible_text` mógł ukryć pełny handshake hosta;
- daemon fast path dodawał `chat_bridge`, ale nie pełny `chatgpt_host_bridge`;
- `ModelGuidedResponseSynthesizer` klasyfikował host ChatGPT jak zwykły brak konfiguracji modelu;
- `WarsawClock` miał rozproszone informacje o źródle i zaufaniu czasu, bez jednego kontraktu środowiska, UTC/local ISO i świeżości.

## Miejsca błędu i zmiany

- `latka_jazn/core/chat_command_contract.py`
  - `write_chat_bridge_payload(...)` nie redukuje już wyniku do tekstu, gdy `host_must_generate_visible_reply=True`;
  - w takim przypadku zwraca pełną pojedynczą linię JSONL z `chatgpt_host_bridge` i `chat_bridge_output.effective_mode=jsonl_host_bridge_envelope`.
- `main.py`
  - `_try_chat_gpt_one_shot_via_daemon(...)` buduje ten sam `build_chatgpt_host_bridge_turn_contract(...)`, co zwykła ścieżka JSONL;
  - `chat_bridge` i zgodnościowy `chatgpt_bridge` pozostają dostępne.
- `latka_jazn/core/model_guided_response_synthesizer.py`
  - `chatgpt_runtime_adapter` zwraca diagnozę `host_chatgpt_bridge_requires_external_visible_reply` i status `host_visible_generation_requested`;
  - `null_model_adapter` ma osobną diagnozę `null_model_adapter_has_no_generation_capability`;
  - rzeczywiste skonfigurowane adaptery zachowują dotychczasową próbę syntezy model-guided.
- `latka_jazn/model_adapters/chatgpt_runtime_adapter.py`
  - status jawnie raportuje `host_visible_generation_required`, `host_must_generate_visible_reply` i fazę host bridge;
  - `can_generate_model_guided_speech=False` oraz `can_attempt_model_guided_speech=False` nie zostały zmienione.
- `latka_jazn/core/clock.py`
  - dodano centralny `TimeSourceResolver` i wynik `TimeSourceResolution`;
  - `WarsawClock` deleguje rozpoznanie środowiska, źródła, świeżości, strefy i nagłówka do resolvera;
  - zachowano stare pola `source`, `trusted`, `sample_iso` dla zgodności.

## Truth boundary po naprawie

1. Lokalny runtime nadal odpowiada za intencję, routing, politykę pamięci, walidację, timestamp i provenance.
2. `chatgpt_runtime_adapter` nie wykonuje lokalnego wywołania ChatGPT i nie udaje model-generated speech.
3. Gdy runtime ma zaakceptowany nietemplate'owy tekst handlera, może go przepuścić jako `chatgpt_host_visible_bridge.accepted=True` i `fallback_classification=not_fallback`.
4. Gdy tekst wymaga języka hosta, kontrakt ma `phase=host_visible_generation_requested` i `host_must_generate_visible_reply=True`.
5. Template, brak walidacji, brak wymaganych komponentów lub pusty handler nadal nie przechodzą jako dynamiczny głos Łatki.
6. `null_model_adapter` nadal pokazuje jawny fallback i nie udaje Łatki ani modelu.

## `--chat-gpt` i final-only

`python -X utf8 main.py --chat-gpt -- "Hej"` nadal jest wygodnym one-shotem. Jeśli runtime ma prawdziwy final, tryb tekstowy zachowuje dotychczasowe zachowanie. Jeśli potrzebna jest faza hosta, tryb tekstowy jest automatycznie promowany do maszynowo czytelnej koperty JSONL. Dzięki temu tekst `RuntimeTurnTruthGate` nie może zostać pokazany jako właściwa odpowiedź rozmowna przy wymaganej generacji hosta.

Test prawdziwej ścieżki `main.main(["--root", <temp>, "--chat-gpt", "--", "Hej"])` potwierdza pełny `chatgpt_host_bridge` bez zapisu do produkcyjnej pamięci/runtime.

## Daemon fast path

Po `chat_daemon(...)` wynik otrzymuje:

- `chat_bridge` z `daemon_fast_path=True`;
- zgodnościowy `chatgpt_bridge`;
- `chatgpt_host_bridge` z tym samym builderem co JSONL bridge;
- ochronę final-only przed ukryciem `host_visible_generation_requested`.

## Resolver czasu

`TimeSourceResolver` używa aware `datetime` i `datetime.now(timezone.utc)` jako lokalnego punktu odniesienia. Konwersja użytkownika używa `ZoneInfo("Europe/Warsaw")`. Zależność `tzdata` pozostaje zadeklarowana w `pyproject.toml`; gdy IANA tzdb nie jest dostępna, resolver zwraca kontrolowany `active_degraded` z `timezone_status=degraded_fixed_offset`, a nie wyjątek startowy.

Kontrakt raportuje:

- `platform_system` z `platform.system()` i `os_name`;
- bezpiecznie wykryty `shell` (`powershell`, `cmd`, `bash`, inne albo `unknown`) i `terminal`;
- `timestamp_source`: `network`, `system_local`, `system_utc`, `runtime_fallback` albo `unavailable`;
- `timestamp_source_detail`, `timestamp_trusted`, `timestamp_freshness_ok`, `timestamp_freshness_seconds`;
- `timezone_key`, `utc_iso`, `local_iso`, `human_time_header`;
- `status`, `timezone_status` i `degradation_reason`.

Brak network time nie zatrzymuje runtime przy świeżym aware czasie lokalnym. Czas lokalny pozostaje `timestamp_trusted=False` / `local_machine_unverified`; nie jest promowany do `trusted_network`. Aktywność daemonu i zaufanie czasu pozostają osobnymi osiami.
