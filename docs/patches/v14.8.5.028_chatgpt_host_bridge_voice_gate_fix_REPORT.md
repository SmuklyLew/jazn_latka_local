# v14.8.5.028 — chatgpt host bridge / voice gate fix

## Cel
Naprawa trasy `--chat-gpt`, która przy poprawnym `chatgpt_runtime_adapter` i poprawnym handlerze rozmownym wpadała w `cannot_answer_directly / model_guided_speech_required` zamiast przepuścić zwalidowany tekst runtime przez widzialny kanał hosta ChatGPT.

## Zmienione pliki
- `latka_jazn/core/engine.py`
- `latka_jazn/core/runtime_answer_validator.py`
- `tests/test_v1485017_model_adapter_contracts.py`

## Zmiana logiczna
1. `RuntimeAnswerValidation` dostał jawny kontrakt `accepted` oraz pole `accepted` w `to_dict()`.
2. `engine.py` rozpoznaje teraz `chatgpt_runtime_adapter` jako jawny host-visible bridge, a nie lokalny model.
3. Jeżeli handler runtime jest zwalidowany, nietemplate'owy, ma komplet wymaganych komponentów i adapter to `chatgpt_runtime_adapter`, odpowiedź przechodzi jako `not_fallback` z `chatgpt_host_visible_bridge.accepted=true`.
4. `origin_truth_valid` akceptuje dwa uczciwe przypadki:
   - lokalna generacja model-guided (`model_generated=true`),
   - zwalidowany runtime handler przepuszczony przez host ChatGPT (`chatgpt_host_visible_bridge.accepted=true`).
5. Test statusu adaptera nie ma już wpisanej na sztywno starej wersji `v14.8.5.026B`, tylko porównuje z `JaznConfig().version`.

## Granica prawdy
Patch nie udaje, że lokalny Python wywołuje ChatGPT jako funkcję. `chatgpt_runtime_adapter` nadal ma `can_generate_model_guided_speech=false`; zmienia się tylko to, że poprawna, zwalidowana odpowiedź handlera runtime może zostać użyta jako finalny widoczny tekst w kanale `--chat-gpt`.
