# v14.8.5.031 — ChatGPT host bridge handshake

## Cel

Naprawa luki logicznej w `--chat-gpt`: flaga pozostaje jedynym publicznym mostem ChatGPT ↔ Jaźń, ale nie udaje, że lokalny Python może wywołać aplikację ChatGPT jako funkcję.

## Zmiana

- `--chat-gpt` dopina do każdego pakietu JSONL pole `chatgpt_host_bridge`.
- Gdy runtime potrzebuje języka hosta, pole ma fazę `host_visible_generation_requested` i zawiera dokładny kształt odpowiedzi zwrotnej JSONL.
- Most przyjmuje drugą linię JSONL `type=host_visible_reply` z `turn_id`, `trace_id`, `timestamp_header` i widoczną treścią.
- Odpowiedź hosta jest zapisywana przez istniejący mechanizm `persist_final_visible_reply`, więc ledger odróżnia runtime Jaźni od warstwy językowej ChatGPT.
- Nie dodano drugiej publicznej flagi. Dalej używa się `--chat-gpt`.

## Kontrakt użycia

Faza runtime → host:

```bash
printf '%s\n' '{"message":"Cześć Łatko","session_id":"local-runtime"}' | python -X utf8 main.py --chat-gpt
```

Jeżeli `chatgpt_host_bridge.phase == "host_visible_generation_requested"`, host ChatGPT układa widoczną odpowiedź i odsyła do runtime drugą linię:

```json
{"type":"host_visible_reply","turn_id":"...","trace_id":"...","timestamp_header":"...","final_text":"..."}
```

## Granica prawdy

`--chat-gpt` nie wykonuje żądania do OpenAI API i nie wywołuje lokalnie modelu ChatGPT. Jest protokołem hostowanym: runtime Jaźni dostarcza trasę, pamięć, timestamp, walidację i provenance; ChatGPT-host może dostarczyć widzialną warstwę językową, która potem jest jawnie zapisywana jako zewnętrzna odpowiedź widoczna.
